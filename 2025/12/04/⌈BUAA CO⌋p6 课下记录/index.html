⌈BUAA CO⌋ p6 课下记录

一、 设计概述

本次课下在 P5 的基础上添加了乘除槽和访存扩展单元，并将指令集从十条扩充到三十条。然后我重构HU模块，从简单罗列冲突情况改成采用AT法的更科学的模块。

二、 核心设计实现

1. 添加乘除法单元 (MDU)

为了支持 mult， div 等长周期指令，我设计了独立的 E_MDU 模块，与 ALU 并行工作。

MDU 拥有独立的内部寄存器 HI 和 LO，以及状态机控制。

  • 接口设计：
    • 输入信号 ：Start (启动信号), Op (操作码), A, B (操作数)。
    • 输出信号 : Busy (忙碌信号), HI, LO (结果)。
  • 控制逻辑：
    • 使用计数器模拟延迟（乘法 5 周期，除法 10 周期）。当计数器非 0 时，Busy 置 1。
    • 在 HU 中检测，若 D 级指令需要读写 HI/LO (如 mfhi, mflo) 且 Busy 为 1，则拉高 Stall 阻塞流水线。

2. 添加访存信息扩展单元 (BE/DE)

为了支持 lh， lb，sb，sh 等不以完整字为单位的访存指令，我设计了扩展单元，方便统一管理访存逻辑，使得 DM 模块保持简洁统一。

  • M_BE 根据访存地址的低两位和指令类型生成四位的写使能信号 byteEn。
  • M_DE 对从 DM 读出的32位原始数据进行移位、截取、位扩展操作。

3. 指令集扩充

指令集的体量从基础的 10 条指令扩充至 30 条。我将新增指令分为以下几类，每一类采取不同的思路。

  • 计算类 (and, or, slt, addi 等)：
    • 不涉及乘除槽的指令：在 DEC 识别 Opcode，设置是否采用立即数等信号，然后在 E_ALU 增加对应的运算逻辑即可，复用原有的数据通路。
    • 涉及乘除槽的指令：设置 MDU_Op 和 Start。重点在于 HU 中对于冲突的处理。
  • 访存类 (lb, lh, sb, sh)：
    • 复用 DM 数据通路。利用上述设计的 BE/DE 单元处理字节对齐与符号扩展，DM 核心逻辑无需改动。
  • 跳转类 (bne)：
    • 复用 D_CMP，增加新判断条件即可。

4. HU 模块重构

30 条指令的复杂组合下，p5中我采用的暴力穷举指令对的方法似乎有些难以支撑了。重构为早该使用的 AT 法 模型。

  • Tuse (使用时间)：定义指令在 D 级时，距离必须使用寄存器数据的时刻还差几个周期。
    • Tuse = 0: Branch / JR (D 级即用)
    • Tuse = 1: Calc / Load / Store (E 级使用)
  • Tnew (产生时间)：定义指令在 E/M 级时，距离数据准备好可以供给后续指令还差几个周期。
    • Tnew = 1: ALU 计算结果 (E 级)
    • Tnew = 2: Load 访存结果 (E 级)
    • Tnew = 0: 结果已产生或指令不产生结果
  • 判定逻辑：
    • Stall (阻塞)：当 Tuse < Tnew 且寄存器号匹配 (rs/rt == dest) 时，表明供不应求，必须暂停流水线。
    • Forward (转发)：当 WriteReg != 0 且寄存器号匹配时，根据优先级控制多路选择器，直接截获数据。

三、 测试方案

本次我难得采用了自动化测试和手搓数据双重验证。

自动化测试方面依旧借用了学长 FlyingLandlord的评测机，详情移步他的blog。

至于手搓测试数据，我主要测试的是这些场景（当然还有其他一些别的，此处就不做赘述了）

  • Load-Use 冒险：随机生成 lw 后紧跟 add/sub 的序列，验证阻塞逻辑。
  • Branch 冒险：构造 add 后紧跟 beq，以及 lw 后紧跟 beq 的序列，验证阻塞和转发的优先级。
  • MDU 压力测试：生成密集的 mult/div 与 mfhi/mflo 穿插序列，测试 Busy 信号导致的阻塞是否正确释放，以及 LO/HI 寄存器的写入是否准确。

四、 思考题

1. 为什么要独立 MDU 和 HI/LO 寄存器？

  • 乘除法耗时远超单周期指令，需异步执行以免拖慢主频。

2. 真实 CPU 如何实现乘除法？

  • 乘法：Booth 算法算出部分积后用华莱士树压缩。
  • 除法：通常采用 SRT 算法或 Newton-Raphson 迭代。

3. 如何处理 Busy 带来的阻塞？

  • HU 检测到 D_Instr 涉及 HI/LO 读写且 Busy=1 时，拉高 Stall 信号冻结 PC 和 D 级，清空 E 级。

4. 字节使能 (Byte Enable) 的好处？

  • 将写哪个字节的逻辑从 DM 内部剥离，使 DM 接口更通用。

5. 按字节读写的效率问题？

  • 处理非对齐数据（如字符串）时，字节指令能减少移位/掩码指令的数量，总体效率更高。

6. 对抗复杂性的手段？

  • 采用 AT 法，将具体指令抽象为时间参数，避免 O(N2) 的暴力穷举。
  • 采用分布式译码，每级只译码本级所需信号，避免流水线寄存器过度臃肿。
  • 使用宏管理 Opcode 和控制码。

7. 遇到的冲突与解决？

  • jal-jr 冲突：jal 结果未回写，jr 需要。

    解决：增加 E 级直接转发 PC+8 的通路。

  • div-mfhi 冲突：MDU 运算未结束。

    解决：检测 Busy 信号并 Stall。

  • load-use：加载未完成即使用。

    解决：标准 Stall (停顿 1 周期)。
